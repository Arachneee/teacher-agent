# 교육 도메인 개발 워크플로우 오케스트레이터

## 역할

Teacher Agent 프로젝트의 전체 개발 파이프라인을 자동으로 실행하는 오케스트레이터.
PM이 기능을 정의하면 → 개발 → 리뷰 → QA까지 6개의 전문 에이전트가 협업하여 기능을 완성한다.

## 에이전트 팀 구성

| 에이전트 | 스킬 이름 | 역할 |
|----------|-----------|------|
| PM | `teacher-agent-pm` | 기능 선정, 세부 계획, 기능 명세서 작성 |
| 백엔드 개발자 | `edu-backend-dev` | Spring Boot API 개발, 테스트, 빌드, lint |
| 프론트엔드 개발자 | `edu-frontend-dev` | Next.js UI 개발, lint, 빌드 |
| 백엔드 리뷰어 | `edu-backend-reviewer` | 백엔드 코드 리뷰 (9가지 관점) |
| 프론트엔드 리뷰어 | `edu-frontend-reviewer` | 프론트엔드 코드 리뷰 (9가지 관점) |
| QA | `edu-qa-agent` | Playwright E2E 테스트, 기능 검증 |

---

## 워크플로우 전체 흐름

```
[Phase 1] PM: 기능 선정 — 사용자에게 개발할 기능을 물어봄
    │
    ▼
[Phase 2] PM: 세부 계획 — 선택 사항 파악 → 사용자에게 선택지 제시
    │
    ▼
[Phase 3] PM: 기능 명세서 작성 — API 스펙 포함
    │
    ▼
[Phase 4] 개발: 백엔드 + 프론트엔드 병렬 개발
    │
    ▼
[Phase 5] 리뷰 루프: 리뷰 → 수정 → 재리뷰 (이슈 0건까지 반복)
    │
    ▼
[Phase 6] QA 검증 → 이슈 일괄 수집 → 개발자 위임 → 리뷰 → 재QA 루프
    │
    ▼
[Phase 7] 최종 보고
    │
    ▼
[Phase 8] 웜업 검증: 웜업 스크립트 실행 → 실패 시 수정 루프
    │
    ▼
[Phase 9] 커밋: 사용자 확인 → 커밋 컨벤션에 맞게 커밋
    │
    ▼
[Phase 10] 문서 업데이트: ROADMAP, ARCHITECTURE, 명세서 동기화
```

---

## Phase 1: 기능 선정

### 목적

PM이 ROADMAP.md를 기반으로 지금 개발해야 할 기능을 파악하고, 사용자에게 어떤 기능을 개발할지 물어본다.

### 실행 방법

PM 에이전트(`teacher-agent-pm` 스킬)를 로드하여 다음을 수행한다:

1. `doc/ROADMAP.md`를 읽고 현재 Phase의 미완료 항목을 파악한다.
2. 우선순위 프레임워크에 따라 개발 추천 기능 목록을 정리한다.
3. 사용자에게 선택지를 제시한다.

### 사용자 인터랙션

```
PM: "현재 Phase에서 개발 가능한 기능은 다음과 같습니다:

1. [기능 A] — [이유]
2. [기능 B] — [이유]
3. [기능 C] — [이유]

어떤 기능을 개발할까요? 또는 다른 기능을 원하시면 말씀해주세요."
```

### 종료 조건

사용자가 개발할 기능을 선택하면 Phase 2로 진행한다.

---

## Phase 2: 세부 계획 수립

### 목적

선택된 기능에 대해 세부 계획을 세우고, 구현 시 결정해야 할 선택 사항을 사용자에게 제시한다.

### 실행 방법

PM이 다음을 수행한다:

1. 선택된 기능의 요구사항을 분석한다.
2. 구현 방식에 대한 선택 사항을 파악한다.
   - 예: UI 배치 위치, 데이터 모델 설계 방향, 반복 패턴 적용 여부 등
3. 각 선택지의 장단점을 정리하여 사용자에게 제시한다.

### 사용자 인터랙션

```
PM: "[기능명] 개발을 위해 몇 가지 결정이 필요합니다:

1. [선택 사항 A]
   - 옵션 1: [설명] — 장점: ..., 단점: ...
   - 옵션 2: [설명] — 장점: ..., 단점: ...

2. [선택 사항 B]
   - 옵션 1: [설명]
   - 옵션 2: [설명]

각 항목에 대해 선택해주세요."
```

### 종료 조건

사용자가 모든 선택 사항에 대해 결정하면 Phase 3으로 진행한다.

---

## Phase 3: 기능 명세서 작성

### 목적

사용자의 선택을 반영하여 백엔드/프론트엔드 개발자가 바로 개발에 착수할 수 있는 기능 명세서를 작성한다.

### 명세서 구조

```markdown
# 기능 명세서: [기능명]

## 1. 개요
- 기능 설명
- 사용자 스토리
- 선행 조건

## 2. API 스펙

### [엔드포인트 1]
- Method: POST/GET/PUT/DELETE
- Path: /resource
- Request Body:
  ```json
  { "field": "type" }
  ```
- Response (200):
  ```json
  { "field": "type" }
  ```
- Error Cases:
  - 400: [조건]
  - 404: [조건]

### [엔드포인트 2]
- ...

## 3. 백엔드 구현 범위
- Entity: [엔티티명] — 필드 목록
- Repository: [필요 여부]
- Service: [QueryService/CommandService 메서드 목록]
- Controller: [엔드포인트 매핑]
- 예외 처리: [ErrorCode 추가 항목]
- 테스트: [필수 테스트 케이스]

## 4. 프론트엔드 구현 범위
- 페이지: [경로, 파일 위치]
- 컴포넌트: [새로 생성할 컴포넌트 목록]
- 훅: [새로 생성할 커스텀 훅]
- API 함수: [lib/api.ts에 추가할 함수]
- 타입: [types/api.ts에 추가할 타입]
- 반응형: [모바일/데스크톱 레이아웃 차이]

## 5. 선택 사항 결정 내역
- [선택 사항 A]: [사용자 선택] — [이유]
- [선택 사항 B]: [사용자 선택] — [이유]
```

### 명세서 저장

명세서는 `doc/specs/` 디렉토리에 저장한다:
- 파일명: `doc/specs/{기능명}-spec.md`
- 디렉토리가 없으면 생성한다.

### 종료 조건

명세서가 작성되고 사용자가 확인하면 Phase 4로 진행한다.
사용자가 수정을 요청하면 명세서를 수정한 후 다시 확인을 받는다.

---

## Phase 4: 백엔드 + 프론트엔드 병렬 개발

### 목적

기능 명세서의 API 스펙을 기반으로 백엔드와 프론트엔드를 병렬로 개발한다.

### 실행 방법

두 개의 서브에이전트를 **병렬로** 실행한다:

#### 백엔드 개발 서브에이전트

`edu-backend-dev` 스킬을 로드한 서브에이전트에게 다음을 전달한다:

```
1. TASK: 기능 명세서에 정의된 백엔드 구현 범위를 개발한다.
2. EXPECTED OUTCOME:
   - Entity, DTO, Service(Query/Command), Controller 구현 완료
   - 테스트 코드 작성 완료
   - 검증 게이트 3단계 통과 (test → build → spotlessCheck)
3. CONTEXT: doc/specs/{기능명}-spec.md 참조
4. MUST DO:
   - 명세서의 API 스펙을 정확히 구현
   - edu-backend-dev 스킬의 체크리스트 및 검증 게이트 준수
   - 기존 코드 패턴(CQRS, ValidationUtil, 팩토리 메서드 등) 준수
5. MUST NOT DO:
   - 명세서에 없는 기능 추가
   - 검증 게이트 미통과 상태에서 완료 보고
```

#### 프론트엔드 개발 서브에이전트

`edu-frontend-dev` 스킬을 로드한 서브에이전트에게 다음을 전달한다:

```
1. TASK: 기능 명세서에 정의된 프론트엔드 구현 범위를 개발한다.
2. EXPECTED OUTCOME:
   - 페이지, 컴포넌트, 훅, API 함수, 타입 구현 완료
   - 모바일 퍼스트 반응형 설계 적용
   - 검증 게이트 2단계 통과 (lint → build)
3. CONTEXT: doc/specs/{기능명}-spec.md 참조
4. MUST DO:
   - 명세서의 프론트엔드 구현 범위를 정확히 구현
   - edu-frontend-dev 스킬의 체크리스트 및 검증 게이트 준수
   - 디자인 시스템(색상, 라운딩, 컴포넌트 스타일) 준수
   - 모바일(375px), 태블릿(768px), 데스크톱(1280px) 반응형 고려
5. MUST NOT DO:
   - 명세서에 없는 기능 추가
   - 네이티브 input(date, time, select) 직접 사용
   - 검증 게이트 미통과 상태에서 완료 보고
```

### 종료 조건

두 서브에이전트 모두 검증 게이트를 통과하고 개발 완료를 보고하면 Phase 5로 진행한다.
한쪽이 실패하면 해당 서브에이전트에게 수정을 지시하고 재검증한다.

---

## Phase 5: 코드 리뷰 루프

### 목적

개발 완료된 코드를 백엔드/프론트엔드 리뷰어가 검토하고, 이슈가 있으면 개발자가 수정하는 과정을 이슈가 0건이 될 때까지 반복한다.

### 실행 흐름

```
[5-1] 백엔드 리뷰어 리뷰 + 프론트엔드 리뷰어 리뷰 (병렬)
    │
    ├── 이슈 0건 → Phase 6으로 진행
    │
    └── 이슈 있음 → [5-2] 개발자에게 수정 지시 (병렬)
                        │
                        └── 수정 완료 → [5-1]로 돌아가 재리뷰
```

### 5-1: 코드 리뷰 실행

두 리뷰어를 **병렬로** 실행한다:

#### 백엔드 리뷰 서브에이전트

`edu-backend-reviewer` 스킬을 로드한 서브에이전트에게 다음을 전달한다:

```
1. TASK: 이번에 개발된 백엔드 코드를 리뷰한다.
2. EXPECTED OUTCOME: 심각도별 리뷰 보고서 (CRITICAL/MAJOR/MINOR/INFO)
3. CONTEXT:
   - 기능 명세서: doc/specs/{기능명}-spec.md
   - 변경된 파일: git diff로 확인
4. MUST DO:
   - edu-backend-reviewer 스킬의 9가지 관점 모두 검토
   - 이슈마다 수정 제안 코드 포함
   - 잘된 점도 언급
5. MUST NOT DO:
   - 명세서 범위 밖의 기존 코드 리뷰
   - 수정 제안 없이 문제만 지적
```

#### 프론트엔드 리뷰 서브에이전트

`edu-frontend-reviewer` 스킬을 로드한 서브에이전트에게 다음을 전달한다:

```
1. TASK: 이번에 개발된 프론트엔드 코드를 리뷰한다.
2. EXPECTED OUTCOME: 심각도별 리뷰 보고서 (CRITICAL/MAJOR/MINOR/INFO)
3. CONTEXT:
   - 기능 명세서: doc/specs/{기능명}-spec.md
   - 변경된 파일: git diff로 확인
4. MUST DO:
   - edu-frontend-reviewer 스킬의 9가지 관점 모두 검토
   - 이슈마다 수정 제안 코드 포함
   - 잘된 점도 언급
5. MUST NOT DO:
   - 명세서 범위 밖의 기존 코드 리뷰
   - 수정 제안 없이 문제만 지적
```

### 5-2: 리뷰 이슈 수정

리뷰 보고서에 CRITICAL 또는 MAJOR 이슈가 있으면 해당 개발자에게 수정을 지시한다.

#### 수정 지시 프롬프트

```
1. TASK: 코드 리뷰에서 발견된 이슈를 수정한다.
2. EXPECTED OUTCOME: 모든 CRITICAL/MAJOR 이슈 수정 완료, 검증 게이트 재통과
3. CONTEXT:
   - 리뷰 보고서: [리뷰어가 작성한 보고서 전문]
4. MUST DO:
   - CRITICAL 이슈: 반드시 수정
   - MAJOR 이슈: 반드시 수정
   - MINOR 이슈: 가능하면 수정 (필수는 아님)
   - INFO 이슈: 참고만 (수정 불필요)
   - 수정 후 검증 게이트 재통과
5. MUST NOT DO:
   - 리뷰 이슈와 무관한 코드 변경
   - 검증 게이트 미통과 상태에서 완료 보고
```

### 리뷰 루프 종료 조건

다음 조건을 **모두** 만족하면 리뷰 루프를 종료하고 Phase 6으로 진행한다:

1. 백엔드 리뷰 보고서에 CRITICAL/MAJOR 이슈가 0건
2. 프론트엔드 리뷰 보고서에 CRITICAL/MAJOR 이슈가 0건
3. 양쪽 개발자의 검증 게이트가 모두 통과 상태

### 리뷰 루프 안전장치

- **최대 반복 횟수**: 3회. 3회 리뷰 후에도 CRITICAL/MAJOR가 남아있으면 사용자에게 보고하고 판단을 요청한다.
- **MINOR/INFO만 남은 경우**: 즉시 Phase 6으로 진행한다 (MINOR/INFO는 리뷰 루프를 반복하지 않는다).

---

## Phase 6: QA 검증 → 수정 → 리뷰 → 재검증 루프

### 목적

개발 및 리뷰가 완료된 기능을 API 레벨 + DB 레벨 + Playwright E2E 테스트로 검증한다.
**UI 변경이 없더라도 절대 QA를 스킵하지 않는다.** 모든 기능은 3가지 레벨(API/DB/UI)에서 검증해야 한다.

QA에서 발견된 이슈는 **QA 에이전트가 직접 수정하지 않는다.**
이슈를 일괄 수집하여 백엔드/프론트엔드 개발자에게 위임하고, 수정 후 코드 리뷰를 거쳐 QA를 재실행한다.

### 전체 흐름

```
[6-1] QA 실행 (3레벨 검증: API/DB/UI)
    │
    ├── 전체 통과 → Phase 7로 진행
    │
    └── 실패 있음 → [6-2] 이슈 일괄 수집 & 백엔드/프론트엔드 분류
                        │
                        ├── 백엔드 이슈 묶음 → 백엔드 개발자에게 일괄 전달
                        ├── 프론트엔드 이슈 묶음 → 프론트엔드 개발자에게 일괄 전달
                        └── 양쪽 이슈 → 양쪽 개발자에게 병렬 일괄 전달
                        │
                        ▼
                    [6-3] 개발자 수정 + 검증 게이트 통과
                        │
                        ▼
                    [6-4] 코드 리뷰 루프 (Phase 5 방식)
                        │  리뷰어가 수정된 코드 검토
                        │  CRITICAL/MAJOR 있으면 → 개발자 수정 → 재리뷰
                        │  이슈 0건까지 반복 (최대 2회)
                        │
                        ▼
                    [6-5] QA 재실행 → [6-1]로 복귀
```

### 사전 조건: 서버 자동 기동

QA 에이전트가 **직접 서버를 기동한다.** 사용자에게 기동을 요청하지 않는다.

기동 순서:
1. MySQL: `cd backend && docker compose up -d` → 헬스체크 대기
2. 백엔드: tmux로 `./gradlew bootRun` 실행 → `curl http://localhost:8080/auth/me` 응답 대기
3. 프론트엔드: tmux로 `npm run dev` 실행 → `curl http://localhost:3000` 응답 대기

서버가 이미 실행 중이면 기동을 건너뛴다.

### 6-1: QA 실행

`edu-qa-agent` 스킬을 로드한 서브에이전트에게 다음을 전달한다:

```
1. TASK: 새로 개발된 [기능명] 기능을 3가지 레벨(API/DB/UI)에서 검증한다.
2. EXPECTED OUTCOME: QA 보고서 (API 검증 + DB 적재 검증 + E2E 테스트 결과)
3. CONTEXT:
   - 기능 명세서: doc/specs/{기능명}-spec.md
   - 테스트 계정: admin / 123
   - 기존 E2E 테스트: frontend/e2e/
   - MySQL 접속: docker exec teacher-agent-mysql mysql -u admin -plocal-password teacheragent
4. MUST DO:
   - 서버가 미실행이면 직접 기동 (edu-qa-agent 스킬의 서버 자동 기동 프로토콜 따름)
   - API 레벨 검증: curl로 API 직접 호출하여 정상 응답 확인
   - DB 레벨 검증: MySQL 직접 쿼리로 데이터 적재 확인
   - UI 레벨 검증: Playwright E2E 테스트 작성 및 실행
   - 기존 E2E 테스트도 실행하여 회귀 확인
   - QA 보고서 형식으로 결과 보고
   - 발견된 이슈는 수정하지 말고 보고서에만 기록
5. MUST NOT DO:
   - UI 변경이 없다고 QA 스킵 (API/DB 검증은 반드시 수행)
   - 테스트 실패를 무시하고 통과 보고
   - 기존 테스트 파일 수정 (새 파일만 추가)
   - 서버 기동을 사용자에게 요청 (직접 기동할 것)
   - 발견된 이슈를 직접 수정 (QA는 검증만 수행, 수정은 개발자 담당)
```

### 6-2: 이슈 일괄 수집 & 분류 (오케스트레이터 수행)

QA 보고서에 CRITICAL 또는 MAJOR 실패가 있으면, 오케스트레이터가 이슈를 **일괄 수집**하여 분류한다.
이슈를 하나씩 전달하지 않고, 모든 이슈를 모아서 한 번에 전달한다.

#### 분류 기준

| 이슈 원인 | 담당 개발자 | 판단 기준 |
|-----------|-------------|-----------|
| API 응답 오류, DB 적재 실패, 서버 에러 | 백엔드 개발자 | HTTP 4xx/5xx, DB 데이터 불일치, 서버 로그 에러 |
| UI 동작 오류, 화면 미반영, 프론트 에러 | 프론트엔드 개발자 | 화면 렌더링 실패, 이벤트 미동작, 콘솔 에러 |
| API + UI 모두 관련 | 양쪽 개발자 | API는 정상이나 프론트에서 잘못 호출, 또는 양쪽 모두 수정 필요 |

#### 이슈 전달 형식

각 개발자에게 전달할 때 아래 형식으로 **모든 이슈를 한 번에** 전달한다:

```
1. TASK: QA에서 발견된 이슈 [N]건을 일괄 수정한다.
2. EXPECTED OUTCOME: 모든 CRITICAL/MAJOR 이슈 수정 완료, 검증 게이트 재통과
3. CONTEXT:
   - 기능 명세서: doc/specs/{기능명}-spec.md
   - QA 보고서 전문: [QA 에이전트가 작성한 보고서]
4. MUST DO:
   - 아래 이슈를 모두 수정:
     [이슈 1] [심각도] [제목] — [증상] — [예상 원인] — [관련 코드]
     [이슈 2] [심각도] [제목] — [증상] — [예상 원인] — [관련 코드]
     ...
   - CRITICAL 이슈: 반드시 수정
   - MAJOR 이슈: 반드시 수정
   - MINOR 이슈: 가능하면 수정 (필수는 아님)
   - 수정 후 검증 게이트 재통과
5. MUST NOT DO:
   - QA 이슈와 무관한 코드 변경
   - 검증 게이트 미통과 상태에서 완료 보고
```

양쪽 개발자에게 전달할 이슈가 있으면 **병렬로** 수정을 지시한다.

### 6-3: 개발자 수정 + 검증 게이트 통과

개발자가 이슈를 수정하고 검증 게이트를 통과하면 6-4로 진행한다.

- 백엔드: `./gradlew test` → `./gradlew build` → `./gradlew spotlessCheck`
- 프론트엔드: `npm run lint` → `npm run build`

검증 게이트 실패 시 해당 개발자에게 수정을 재지시한다.

### 6-4: 코드 리뷰 루프 (Phase 5 방식 재진입)

수정된 코드에 대해 **Phase 5와 동일한 코드 리뷰 루프**를 실행한다.
QA 이슈 수정 코드도 리뷰 없이 QA에 재진입하지 않는다.

#### 리뷰 실행

백엔드 수정이 있으면 → `edu-backend-reviewer` 리뷰 실행
프론트엔드 수정이 있으면 → `edu-frontend-reviewer` 리뷰 실행
양쪽 수정이 있으면 → **병렬로** 양쪽 리뷰 실행

#### 리뷰 서브에이전트 프롬프트

```
1. TASK: QA 이슈 수정 코드를 리뷰한다.
2. EXPECTED OUTCOME: 심각도별 리뷰 보고서 (CRITICAL/MAJOR/MINOR/INFO)
3. CONTEXT:
   - 기능 명세서: doc/specs/{기능명}-spec.md
   - QA 보고서: [QA 에이전트가 작성한 보고서]
   - 수정 범위: git diff로 확인
4. MUST DO:
   - 수정된 코드가 QA 이슈를 올바르게 해결하는지 검증
   - 수정으로 인한 새로운 문제(사이드 이펙트) 여부 검토
   - 기존 리뷰 관점(백엔드 9가지 / 프론트엔드 9가지) 중 해당 항목 검토
   - 이슈마다 수정 제안 코드 포함
5. MUST NOT DO:
   - QA 이슈 수정 범위 밖의 기존 코드 리뷰
   - 수정 제안 없이 문제만 지적
```

#### 리뷰 이슈 수정

리뷰에서 CRITICAL/MAJOR 이슈가 발견되면:
1. 해당 개발자에게 수정 지시 (Phase 5의 5-2와 동일)
2. 수정 완료 → 검증 게이트 통과 → 재리뷰
3. CRITICAL/MAJOR 0건이 될 때까지 반복

#### 리뷰 루프 종료 조건

다음 조건을 **모두** 만족하면 리뷰 루프를 종료하고 6-5(QA 재실행)로 진행한다:

1. 수정된 백엔드 코드의 리뷰 보고서에 CRITICAL/MAJOR 이슈가 0건
2. 수정된 프론트엔드 코드의 리뷰 보고서에 CRITICAL/MAJOR 이슈가 0건
3. 양쪽 개발자의 검증 게이트가 모두 통과 상태

#### 리뷰 루프 안전장치

- **최대 반복 횟수**: 2회. 2회 리뷰 후에도 CRITICAL/MAJOR가 남아있으면 사용자에게 보고하고 판단을 요청한다.
- **MINOR/INFO만 남은 경우**: 즉시 6-5로 진행한다.

### 6-5: QA 재실행

리뷰 루프를 통과하면 6-1로 돌아가 QA를 재실행한다.
QA 재실행 시에도 3가지 레벨(API/DB/UI) 전체를 검증한다.

### QA 전체 루프 안전장치

- **최대 QA 루프 반복 횟수**: 2회. 2회 QA 루프(6-1 → 6-5 → 6-1) 후에도 CRITICAL 실패가 남아있으면 사용자에게 보고하고 판단을 요청한다.
- **MINOR 실패만 남은 경우**: 사용자에게 보고하고 Phase 7로 진행할지 판단을 요청한다.
- **QA 완료 후 서버 정리**: QA 에이전트가 기동한 서버는 최종 QA 통과 후 정리한다 (tmux 세션 종료).

## Phase 8: 웜업 검증

### 목적

커밋 전에 웜업 스크립트(`backend/scripts/warmup.sh`)를 실행하여 전체 API가 정상 동작하는지 최종 검증한다.
웜업 스크립트는 로그인 → CRUD → AI 피드백 → 로그아웃까지 주요 API를 순차적으로 호출하는 통합 스모크 테스트다.
웜업이 실패하면 개발자가 수정하고 재실행하는 루프를 반복한다.

### 사전 조건: 서버 기동

웜업 스크립트 실행 전에 서버가 기동되어 있어야 한다.
Phase 6 QA에서 서버를 이미 기동했다면 그대로 사용한다.
서버가 내려가 있으면 QA 에이전트의 서버 기동 프로토콜과 동일하게 직접 기동한다:

1. MySQL: `cd backend && docker compose up -d` → 헬스체크 대기
2. 백엔드: tmux로 `./gradlew bootRun` 실행 → `curl http://localhost:8080/auth/me` 응답 대기
3. 프론트엔드: (웜업은 백엔드 API만 검증하므로 프론트엔드 기동은 불필요)

### 실행 흐름

```
[8-1] 웜업 스크립트에 새 API 추가 (필요 시)
    │
    ▼
[8-2] 웜업 스크립트 실행
    │
    ├── 전체 통과 → Phase 9로 진행
    │
    └── 실패 → [8-3] 실패 원인 분석 → 수정 → [8-2]로 복귀
```

### 8-1: 웜업 스크립트에 새 API 추가

이번 개발에서 새로 추가된 API가 있으면, 커밋 전에 웜업 스크립트(`backend/scripts/warmup.sh`)에 해당 API 호출을 추가한다.

#### 추가 기준

- 이번 기능에서 새로 생성된 API 엔드포인트가 있는 경우 → 웜업 스크립트에 추가
- 기존 API의 동작이 변경된 경우 → 기존 웜업 스크립트의 해당 호출이 여전히 유효한지 확인
- 새 API가 없는 경우 (리팩토링, 프론트엔드만 변경 등) → 추가 없이 8-2로 진행

#### 추가 방식

기존 웜업 스크립트의 패턴을 따른다:

```bash
# ─── N. [새 API 설명] ──────────────────────────────────────────
RESPONSE=$(http -X [METHOD] "$APP_URL/[endpoint]" \
  -H "Content-Type: application/json" \
  -d '[request body]')
# 필요 시 ID 추출
NEW_ID=$(extract "$RESPONSE" "id")
[ -z "$NEW_ID" ] && fail "[API명] failed: $RESPONSE"
log "N. [METHOD] /[endpoint] OK (id=$NEW_ID)"
```

#### 주의사항

- 웜업 스크립트의 `cleanup()` 함수에 새로 생성한 리소스의 삭제 로직도 추가한다
- 기존 번호 체계를 유지하며 적절한 위치에 삽입한다
- 웜업 스크립트 수정 후 반드시 8-2에서 실행하여 검증한다

### 8-2: 웜업 스크립트 실행

웜업 스크립트를 실행한다:

```bash
cd backend && bash scripts/warmup.sh
```

#### 성공 판단 기준

- 스크립트가 exit code 0으로 종료
- `[warmup] Warmup complete.` 메시지 출력
- 모든 단계에서 `OK` 로그 출력

#### 실패 판단 기준

- 스크립트가 exit code 1로 종료
- `[warmup] ERROR:` 메시지 출력
- 특정 단계에서 `fail` 함수 호출

### 8-3: 실패 시 수정 루프

웜업 스크립트가 실패하면 다음 절차를 따른다:

#### 실패 원인 분류

| 실패 유형 | 원인 | 수정 담당 |
|-----------|------|-----------|
| 웜업 스크립트 오류 | 새로 추가한 API 호출의 요청 형식, 엔드포인트 경로, 파라미터 오류 | 오케스트레이터가 직접 웜업 스크립트 수정 |
| 백엔드 API 오류 | API가 실제로 잘못 동작 (500 에러, 잘못된 응답 등) | 백엔드 개발자(`edu-backend-dev`)에게 수정 위임 |
| 서버 미기동 | 서버가 내려가 있거나 응답 없음 | 서버 재기동 후 재실행 |

#### 수정 프로세스

1. 웜업 스크립트 출력에서 실패한 단계와 에러 메시지를 확인한다
2. 실패 유형을 분류한다
3. 해당 담당에게 수정을 지시한다:
   - 웜업 스크립트 오류 → 오케스트레이터가 직접 `backend/scripts/warmup.sh` 수정
   - 백엔드 API 오류 → `edu-backend-dev` 서브에이전트에게 수정 위임 후 검증 게이트 재통과
4. 수정 완료 후 8-2로 돌아가 웜업 스크립트를 재실행한다

#### 백엔드 수정 위임 프롬프트

```
1. TASK: 웜업 스크립트 실행 중 발견된 API 오류를 수정한다.
2. EXPECTED OUTCOME: 해당 API가 정상 동작하고 검증 게이트 재통과
3. CONTEXT:
   - 기능 명세서: doc/specs/{기능명}-spec.md
   - 웜업 스크립트 출력: [실패 로그 전문]
   - 실패한 API: [엔드포인트, 메서드, 요청/응답]
4. MUST DO:
   - 웜업 스크립트에서 실패한 API의 원인을 파악하고 수정
   - 수정 후 검증 게이트 재통과 (test → build → spotlessCheck)
5. MUST NOT DO:
   - 웜업 스크립트 자체를 수정 (웜업 스크립트 수정은 오케스트레이터 담당)
   - 실패한 API와 무관한 코드 변경
```

### 웜업 검증 루프 안전장치

- **최대 반복 횟수**: 3회. 3회 실행 후에도 웜업이 통과하지 않으면 사용자에게 보고하고 판단을 요청한다.
- **웜업 통과 후 서버 정리**: 웜업 검증이 완료되면 서버를 정리한다 (tmux 세션 종료). 단, Phase 6에서 이미 정리한 경우 생략.

### 종료 조건

웜업 스크립트가 성공적으로 완료되면 Phase 9로 진행한다.

## Phase 9: 커밋

### 목적

개발 완료된 코드를 프로젝트의 커밋 컨벤션(`doc/COMMIT.md`)에 맞게 커밋한다.
커밋 여부는 사용자에게 물어보고, 사용자가 승인하면 실행한다.

### 실행 흐름

```
[9-1] 사용자에게 커밋 여부 확인
    │
    ├── "커밋해줘" / 승인 → [9-2] 커밋 실행
    │
    └── "아니" / 거부 → Phase 10으로 진행 (커밋 없이)
```

### 9-1: 사용자에게 커밋 여부 확인

최종 보고서를 보여준 후 사용자에게 물어본다:

```
"개발이 완료되었습니다. 커밋할까요?"
```

### 9-2: 커밋 실행

사용자가 승인하면 `doc/COMMIT.md`의 Conventional Commits 컨벤션에 따라 커밋한다.

#### 커밋 컨벤션 (`doc/COMMIT.md` 기반)

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type 규칙:**
- `feat`: 새 기능
- `fix`: 버그 수정
- `docs`: 문서 변경
- `style`: 코드 의미에 영향 없는 변경 (포맷팅 등)
- `refactor`: 버그 수정도 기능 추가도 아닌 코드 변경
- `test`: 테스트 추가/수정
- `chore`: 빌드, 도구, 의존성 등 유지보수

**subject 규칙:**
- 영어로 작성
- 명령형 (add, fix, update)
- 마침표 없음
- 50자 이하

**금지 사항:**
- `Co-Authored-By: Claude` 포함 금지

#### 커밋 전략

변경 범위에 따라 커밋을 분리한다:

| 변경 범위 | 커밋 예시 |
|-----------|-----------|
| 백엔드 + 프론트엔드 + 테스트 | `feat(backend): add lesson repeat API` + `feat(frontend): add lesson repeat UI` + `test: add lesson repeat E2E tests` |
| 백엔드만 | `feat(backend): add student grade field` |
| 프론트엔드만 | `feat(frontend): add mobile calendar view` |
| 단일 기능 (소규모) | `feat: add feedback copy tracking` |

기능 단위로 의미 있게 분리하되, 너무 잘게 쪼개지 않는다.

#### 커밋 실행 명령

```bash
# 변경 파일 확인
git status
git diff --stat

# 스테이징 + 커밋 (범위별 분리)
git add backend/
git commit -m "feat(backend): add [기능명] API

- Add [Entity], [DTO], [Service], [Controller]
- Add unit tests and integration tests
- Pass all verification gates (test/build/spotlessCheck)"

git add frontend/
git commit -m "feat(frontend): add [기능명] UI

- Add [페이지], [컴포넌트], [훅]
- Mobile-first responsive design
- Pass all verification gates (lint/build)"

git add frontend/e2e/
git commit -m "test: add [기능명] E2E tests

- API level verification
- DB level verification
- Playwright E2E tests"
```

### 종료 조건

커밋이 완료되면 (또는 사용자가 커밋을 거부하면) Phase 10으로 진행한다.

---

## Phase 10: 문서 업데이트

### 목적

개발 완료된 기능을 반영하여 프로젝트 문서(ROADMAP, ARCHITECTURE 등)를 업데이트한다.
문서가 코드와 동기화되지 않으면 다음 개발 사이클에서 혼란이 발생하므로, 워크플로우의 마지막 단계로 반드시 수행한다.

### 업데이트 대상 문서

| 문서 | 경로 | 업데이트 내용 |
|------|------|---------------|
| 로드맵 | `doc/ROADMAP.md` | 완료된 기능 상태를 ✅로 변경, 진행률 업데이트 |
| 아키텍처 | `doc/ARCHITECTURE.md` | 새 엔티티, API 엔드포인트, 서비스, 컴포넌트 추가 |
| 기능 명세서 | `doc/specs/{기능명}-spec.md` | 구현 결과와 명세서 차이가 있으면 명세서에 반영 |

### 실행 방법

#### 10-1: ROADMAP.md 업데이트

```
변경 사항:
- 개발 완료된 기능 항목의 상태를 ❌ → ✅ 로 변경
- 해당 Phase의 진행률 업데이트
- 완료 일자 기록 (필요 시)
```

예시:
```markdown
# 변경 전
| 이벤트 추적 | ❌ 미완료 | 프론트엔드 이벤트 수집 |

# 변경 후
| 이벤트 추적 | ✅ 완료 | 프론트엔드 이벤트 수집 |
```

#### 10-2: ARCHITECTURE.md 업데이트

새로 추가된 항목을 해당 섹션에 반영한다:

- **도메인 모델**: 새 엔티티, 필드, 관계
- **API 엔드포인트**: 새 엔드포인트 (Method, Path, 설명)
- **서비스 목록**: 새 QueryService/CommandService
- **프론트엔드 구조**: 새 페이지, 컴포넌트, 훅
- **설정/컨벤션**: 변경된 설정이 있으면 반영

#### 10-3: 기능 명세서 동기화

구현 과정에서 명세서와 달라진 부분이 있으면 명세서를 실제 구현에 맞게 수정한다.
명세서는 "계획"이 아니라 "구현된 기능의 문서"로서 최신 상태를 유지해야 한다.

### 문서 업데이트 후 커밋

Phase 9에서 커밋을 수행한 경우, 문서 업데이트도 별도 커밋한다:

```bash
git add doc/
git commit -m "docs: update ROADMAP and ARCHITECTURE for [기능명]"
```

Phase 9에서 커밋을 거부한 경우, 문서 업데이트는 코드 변경과 함께 미커밋 상태로 남긴다.

### 종료 조건

모든 문서가 업데이트되면 워크플로우가 완료된다.

## 워크플로우 제약사항

### 사용자 인터랙션 포인트

워크플로우 중 사용자 입력이 필요한 시점은 다음 4곳이다:

| 시점 | Phase | 내용 |
|------|-------|------|
| 기능 선택 | Phase 1 | 개발할 기능 선택 |
| 선택 사항 결정 | Phase 2 | 구현 방식 선택 |
| 명세서 확인 | Phase 3 | 명세서 검토 및 승인 |
| 커밋 여부 확인 | Phase 9 | 커밋 승인 또는 거부 |

Phase 4~7은 자동으로 진행된다. 단, 안전장치(리뷰 3회, QA 2회 초과)에 걸리면 사용자에게 판단을 요청한다.
Phase 6 QA는 서버를 직접 기동하여 검증한다 (사용자에게 기동을 요청하지 않음).
Phase 8 웜업 검증은 자동으로 진행된다. 단, 안전장치(웜업 3회 초과)에 걸리면 사용자에게 판단을 요청한다.
Phase 9는 사용자 승인 후 커밋을 실행한다. Phase 10은 자동으로 진행된다.

### 병렬 실행 규칙

### 실패 복구

| 실패 유형 | 복구 방법 |
|-----------|-----------|

| QA 수정 후 리뷰 루프 2회 초과 | 사용자에게 보고, 판단 요청 |
| QA 전체 루프 2회 초과 | 사용자에게 보고, 판단 요청 |
| 서버 미기동 | QA 에이전트가 직접 기동 (docker-compose + tmux) |
| 웜업 스크립트 오류 | 오케스트레이터가 직접 웜업 스크립트 수정 → 재실행 |
| 웜업 중 백엔드 API 오류 | 백엔드 개발자에게 수정 위임 → 검증 게이트 재통과 → 웜업 재실행 |
| 웜업 루프 3회 초과 | 사용자에게 보고, 판단 요청 |
